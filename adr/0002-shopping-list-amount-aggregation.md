# 买菜清单用量聚合：严格解析、同单位才相加，否则按来源罗列

## Context

配料的用量是自由文本（`2汤匙`、`半茶匙`、`适量`、`2个（约400g）`），无法可靠地一律相加。需要在「智能求和（可能算错）」与「只罗列（永不出错但不省心）」之间取舍。一个「算错的清单」比「不合并的清单」更坑，因为用户会照着它买错量。

## Decision

采用**严格解析 + 安全回退**：仅当整串是「数字（含半 / 小数）+ 已知单位」时才纳入求和，且只有单位完全相同才相加；不做跨单位换算（茶匙 ↔ 汤匙），不解析中文数字与范围（`2-3`）。任何带括号 / `适量` / 未知单位 / 多单位的用量一律回退，按 `{ 用量, 菜名 }` 如实罗列。原则：**宁可少合并，绝不算错。**

## Considered Options

- **智能解析、尽量算**（含跨单位换算、剥括号取主数字、解析中文数字）——被拒：`2个（约400g）` 这类数据极易解析出错，错误的量比不合并更危险。
- **只罗列、完全不加**——被拒：能干净相加的常见情形（`2汤匙 + 1汤匙`）白白放过，不够省心。

## Consequences

- 不提供**份量缩放**（同源问题：用量非结构化，缩放不可靠）。
- 食材归组先过一张**同义词表**做模糊合并（`油 / 色拉油 / 植物油 → 食用油`、`西红柿 → 番茄`，可扩充），表外按精确名分组。刻意只并入明确列出的别名——`香油 / 辣椒油 / 蚝油` 不会被误并进 `油`。同一「宁可少合并绝不误并」的原则。见 [CONTEXT.md](../CONTEXT.md) 中「食材」。
